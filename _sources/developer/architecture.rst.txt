Architecture
============

The present page maps out the solver's source layout and the
principal classes that a developer is likely to encounter when
extending the rovibronic machinery. A more extended walk-through
is provided in
``docs/Source Code Report/rovibronic_dsmc_report.pdf``.

Top-level source layout
-----------------------

.. code-block:: text

   src/                 C++ libraries (DSMC core + rovibronic extension)
   applications/        Solver executable and utilities
   run/                 Reference test cases (Tutorial, verification, …)
   doc/                 Standalone LaTeX notes (derivations, audits)
   docs/                Long-form PDFs and the Sphinx source of this site
   Allwmake             Top-level build driver

The DSMC core
-------------

All Lagrangian particle machinery lives under
``src/lagrangian/DSMC/``:

* ``clouds/dsmcCloud.C`` —  the top-level cloud: particle storage,
  collision-loop driver, reaction orchestration, species-database
  loading. Most dictionary keys are read in this file; in
  particular, the rovibronic-specific switches are read around
  lines 49–92 (``rovibronicDatabase`` and legacy
  ``moleculeProperties`` branches) and 1040–1061 (relaxation
  collision numbers and solver-level switches).
* ``parcels/`` —  per-particle data types, including the quantum
  state triple :math:`(n, v, J)`.
* ``submodels/`` —  VSS scattering, relaxation models, chemistry
  backends, and the spectroscopic database.
* ``boundaries/basic/`` and ``boundaries/derived/patchBoundaries/``
  —  wall, cyclic, and inflow/outflow patch classes.

The rovibronic extension
------------------------

On top of the DSMC core, the rovibronic extension adds:

* the quantum state triple on the parcel (stored alongside
  position, velocity, and weight);
* the Dunham-based construction of the energy and degeneracy
  tables at solver start-up, together with the PEC asymptote-
  matching for dissociation (printed at start-up as a diagnostic
  table, typically with errors below :math:`1~\mathrm{cm}^{-1}`);
* Larsen-Borgnakke sampling on the quantised state space, rather
  than on a continuous internal-energy variable, with the
  parallel-marginal cascade logic described in
  :doc:`../theory/relaxation`;
* automatic heats-of-reaction computation for dissociation,
  recombination, and exchange, driven by the database entries
  rather than by user input.

Rovibronic-aware boundary
`````````````````````````

The patch class ``dsmcMixedDiffuseSpecularWallRotPatch``
(``src/lagrangian/DSMC/boundaries/derived/patchBoundaries/dsmcMixedDiffuseSpecularWallRotPatch/``)
is the rovibronic-aware wall patch: on diffuse reflection the
internal state :math:`(n, v, J)` is re-sampled from the wall
temperature using the database partition functions. Other patch
classes (``dsmcDiffuseWallPatch``, ``dsmcSpecularWallPatch``,
``dsmcCLLWallPatch``, and the various ``dsmc…InflowPatch``
variants) are unaware of the quantum-state bookkeeping and
sample the internal state from the stock continuous model.

Species database
````````````````

Per-species level tables are loaded from
``constant/speciesCache/<species>.dat`` at run-time. The
reference tutorial ships 259 such files, covering diatomic
molecules of air chemistry, their atomic fragments, the first
few ion species, and the electron. Replacing a species in a
running case therefore only requires replacing its ``.dat`` file
and editing the ``typeIdList``.

Random-number generator
-----------------------

A single ``Foam::Random`` instance, ``rndGen_``, is held on the
cloud (``dsmcCloud.H``, member variable). It is constructed at
``dsmcCloud.C:1122`` with the seed

.. code-block:: cpp

   rndGen_(label(clock::getTime()) + 7183*Pstream::myProcNo())

i.e. the wall-clock time offset by a per-rank multiple of 7183.
All downstream RNG consumers —  Larsen-Borgnakke sampling,
wall-interaction sampling, parcel initialisation, and the
Maxwellian resampling controller —  access this single instance
through ``cloud_.rndGen()``.

It is important to emphasise that the **single entry point makes
deterministic regression testing feasible**: routing a fixed seed
into the ``Random`` constructor at this one location seeds every
downstream RNG consumer automatically. The design of the planned
regression-test infrastructure is documented in
:doc:`regression_tests`.
